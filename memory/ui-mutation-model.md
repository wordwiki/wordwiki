The rabid/liminal UI design, kept deliberately simple:

- **Read side:** pages present information, formatted as richly as needed. A *page* route (navigable entry point) returns `templates.page(title, body)` (a marked `{[pageMarker], title, body}`) rather than assembling a document; `Rabid.rpcHandler` wraps it in `pageTemplate` for a top-level navigation, or returns just the body plus a `<title>` element (so htmx updates the tab) when the request carries `HX-Request`. Routes that return plain markup are *fragments* and are never wrapped — that's the common case (they replace part of a page via htmx swaps). So a page route serves both full-nav and htmx-swap correctly with one definition.
- **Write side:** *every* mutation takes one of three forms —
  1. a single button that performs an action immediately;
  2. a button that asks for confirmation, then performs the action;
  3. a button that pops a simple modal collecting the **arguments to the action**, then performs it when Save is clicked.

Key point about (3): the modal is general — it collects the **arguments to a call**, not merely the fields of a record. Editing a volunteer (a Bootstrap modal with the row's fields pre-populated) is the *special case* where the arguments are a row's columns. The general case is e.g. a "bike sales in a date range" query that pops a start/end-date form, then runs the query. So all three forms unify as "an action with a parameter list" (immediate = 0 params; confirm = 0 params + gate; modal = N params).

**Why:** one interaction vocabulary for every state change keeps the whole system small and predictable.

**How to apply:** build mutations only in these three forms. Treat the `Field` machinery as *parameter widgets*, not just record columns.

The popup (form 3) is **two round-trips, both through the normal expr/route mechanism**: (1) the page button submits some parameters (baked into its route expression) to a server function that *generates* the dialog using the same `Field` widgets as the tables — the dialog may include **hidden fields** for parameters that are fixed by whichever button opened it (not user-editable) but must still ride along on submit; (2) submitting the dialog performs the action with the combined visible + hidden values. Record-edit is exactly this: edit button → `renderForm(getById(id))` (generator; the primary key is a hidden field) → `saveForm(getFormJSON(...))` (action).

Current primitives (verify before relying — true as of 2026-06): `liminal/action.ts` provides `actionButton()` (immediate/confirm/modal) and `renderParamForm(params, defaults, {hidden, dispatch})` — the generalization of the old `TableEditForm` (now deleted) from record columns to call arguments. **Record editing is consolidated onto this**: `Table.renderForm` builds the edit form via `renderParamForm`, passing the primary key and a `before-<field>` snapshot of each editable field as *hidden* params (so the before-values — the lock-free conflict protection — never appear in the non-record param case). `renderInput` no longer emits `before-` fields; the record editor supplies them. `liminal/table.ts` `Field` subclasses provide `renderInput`/`parseInput`; `editButtonProps` + `#modalEditor` + `showModalEditor()` (in `rabid/templates.ts` / `resources/rabid-scripts.js`) load a form into the modal. A dialog's `dispatch` attrs decide what its submit does: a mutation uses `onsubmit="tx`expr(getFormJSON(event.target))`"` → server returns `{action:'reload'|'alert', targets}` (interpreted by `tx()`); a query/view uses `hx-get`+`hx-target` to swap the rendered result into a container (see `rabid/volunteer.ts` `searchDialog`/`searchResults`).

**Record CREATE (2026-06, committee is the worked example):** a "New X" button is just form (3) with the record form rendered over an *empty* record — `newDialog()` → `renderForm({} as T)` (no primary key → hidden pk absent) → `saveForm` takes the INSERT branch and reloads `.-<table>-` (so the list wrapper must be a reloadable fragment tagged with the pk-less class, e.g. `committee.renderCommitteeList()`). `Table.saveForm`'s insert branch was dead code until 2026-06 (`parseFormWithPrimaryKey` threw on missing pk — now uses `parseForm`). Server-side gate: `renderForm` refuses via `recordEdit` on the empty record. Owned child rows a record needs at birth (e.g. a committee's `volunteer_group`) are supplied by overriding `Table.insert` — works for both direct calls and the saveForm path; the FK field hides from the form via `isVisible() false` (see `OwnedGroupField` in `rabid/group.ts`), which also exempts it from required-input validation.

**Prefilled CREATE (2026-06, "New task in this project" is the worked example):** a new-record dialog may render over a *partial* record (`renderForm({project_id} as Task)`) to preset fields. This works because `renderForm` emits **empty `before-` snapshots when there is no pk** — on an insert every supplied value is a change; snapshotting the prefill would silently drop any preset the user accepts unchanged. Columns managed entirely by table code (order keys, change stamps) use the same `isVisible() false` trick as `OwnedGroupField` and are supplied by `insert()`/`update()` overrides (see `ManagedStringField`/`ManagedDateTimeField` in `rabid/task.ts`; `TaskTable` stamps `last_change_time` in `update`/`updateNamedFields` overrides, and subtask mutations `touch()` their parent task).

**Presentation principle (2026-06-12, dz):** pages should read as **documents** — beautiful presentations of the information (e.g. a project's ongoing state), *coincidentally* editable via quiet line affordances. Casual volunteers are the primary users: they won't learn the system, and unlike corporate software they can just stop using it if it confuses them. Concretely: no card framing, no empty-state filler text ("No checklist items." is noise — absence IS the state), no permanent button rows; common verbs = checkbox/pencil; less-common verbs go one tap behind a **☰ menu** (`action.actionMenu` in liminal/action.ts — Bootstrap dropdown, delegation-based so htmx-swapped fragments need no re-init), IN PLACE rather than on detail pages casual users won't find. Future: move up/down join the ☰; badge/pencil density is a pending tuning pass. Assigned-to is DONE slimming (2026-06-12): one wrapping line everywhere — `Assigned to:` + chips + inline dashed `+ Add member` chip (`.lm-chip-add`, lives INSIDE the membership fragment so reloads keep it) + ☰ for committee/customize/revert; explainer sentences and "No members yet." dropped.
