Rabid tests run the app **as a library** (no HTTP, no browser): set an actor, render a route to markup, fire an action, render again. State lives in SQL and pages are pure functions of state, so this covers most logic. Tests use a fresh **in-memory** SQLite (`setDefaultDb`/`Db.openMemory` in `liminal/db.ts`) and never touch `database/db.db` — so a running rabid is irrelevant to tests (the single-instance/SQLite-writer concern does not apply to the test path).

Layers (all framework-agnostic so a move off Deno is a one-file change):
- `liminal/testing/test.ts` — import `{ test }` from here, **NOT `Deno.test`** (this is the only file that names Deno).
- `liminal/testing/assert.ts` — single import choke-point for assertions.
- `liminal/testing/markup-assert.ts` — query the `[tag, attrs, ...children]` arrays: `getByTestId`, `findAll`, `text`, `hasText` (assert on the markup, not serialized HTML).
- `liminal/testing/db-harness.ts` — generic `openTestDb`/`clearAllData`.
- `rabid/testing.ts` — `withTestDb(fn)` (in-memory db + reset + seed fixture, passes handles), `asUser(id,fn)`/`asAnon`/`asSystem` (ambient [[field-security-model]] context), `renderRoute(path)` (returns markup; a page() is unwrapped to its body), `invoke(path, ...args)` (rpc action; throws on server error so use assertRejects).
- `rabid/test-fixtures.ts` — `buildFixture`: alice(host), bob(shares contact), carol(private), dave(admin).

`data-testid` hooks in views are structural only — never embed a protected value (would defeat redaction). Run: `deno test --allow-all rabid/volunteer_test.ts`.

Fake data (`rabid/fake_data.ts`) is separate (realistic *volume*, not assertion fixtures): per-concern seeded faker streams (`makeStreams`) make it reproducible and stable (adding a column doesn't churn others); exported, parameterized builders (`seedVolunteers`/`seedEvents`/`seedEventCommitments`/`seedTimesheets`) composed by named scenarios. `./create_fake_data.sh <scenario>` (default `dev`; scenarios: minimal/dev/full/activityReport). Timesheets are the expensive slice — only `full`/`activityReport` generate them.

Destructive-op guard: a framework `config` table (`rabid/config.ts`) holds a `db_purpose` marker that travels with the data. `assertSafeToWipe` **refuses to wipe a db marked `production`** (warns on an unmarked one); fake-data rebuilds mark the throwaway db `dev`. Protection is opt-in — a real database MUST be marked via `deno run rabid/fake_data.ts set_db_purpose production` to be protected. (Note: an external `set_db_purpose`/schema-rebuild writer blocks on SQLite's lock while a server holds the db — stop the server first.)

The generic server/framework core now lives in **`liminal/liminal.ts` (`abstract class LiminalApp`)** — route dispatch, the HTTP/rpc handler, server lifecycle, the browser-test bridge, the named-run launcher, and `/eval`. **`class Rabid extends LiminalApp`** (`rabid/rabid.ts`) supplies hooks: `appName`, `routes`, `resolveSecurityContext`, `getDbPurpose`, `stamp/mostRecentTestClient`, `makePage`, `resourceContentDirs`, `testRuns`, and overrides `coercePageResult`/`rewriteUnauthenticatedRoute`/`evalServer` (the last so server-`/eval` sees rabid's lexical scope — direct eval only reaches names at its definition site). **`class WordWiki extends LiminalApp` too** (`wordwiki/wordwiki.ts`, served from `~/mmo` on :9000, home at `/ww/`): overrides `routeSegment`→'ww' (URL prefix differs from scope name 'wordwiki'), `homeRouteExpr`→'wordwiki.home()', `requestHandlerPaths` (/ww/, /page/, /workspace-rpc-and-sync), anon `resolveSecurityContext`, in-memory test-client identity (no sessions yet). Base split `routeSegment` (URL segment) from `appName` (scope name) and added a `query` alias (= queryArgs) for it. `deno-http-server` routes by **longest-matching prefix** across content dirs + handler mounts (a catch-all `'/'`→`'./'` content dir no longer shadows the dynamic `/ww/` handler). Run wordwiki: `./publishHomeAndServe.sh` (or skip transpile: `cd ~/mmo && deno run -A ~/wordwiki/wordwiki/wordwiki.ts serve`).

Browser-test bridge — **full reference doc: `liminal/docs/browser-testing.md`** (run JS in a REAL browser from server-side test code, no puppeteer/CDP): a logged-in browser with the `testing` permission opens `/rabid/rabid.testClientPage()` (explicit opt-in) — `resources/test-agent.js` opts in, then long-polls for commands, evals each as an async-fn body, structured-serializes the result (tags `undefined`/`NaN`/`Date`/DOM nodes/cycles), and posts it back. Server-side `rabid.evalInBrowser(js)` pushes a command and awaits the value. Transient channel = `liminal/browser-agent.ts` (in-memory, keyed by session_token, one command in flight at a time); durable identity/liveness = session columns `last_test_client_opt_in` / `last_test_client_heartbeat`. Target is **always the most-recent opt-in, never an older one** (deterministic; heartbeat only sharpens the timeout error). Gated twice: `testing` permission + refused on a `production` db. Dev login Rocky (`rocky@redraccoon.org`/`rcky`) now has `admin,testing`. Fake data also seeds canonical fixed logins per role tier for checking role-dependent UI by hand: `hazel@redraccoon.org`/`hzl` (host), `vinnie@redraccoon.org`/`vnny` (regular, no roles).

**Puppeteer login: ALWAYS use the one-navigation GET login** (non-production dbs only — never fill the login form field-by-field, it wastes several round-trips and races navigation): navigate to `http://localhost:8888/rabid.loginRequest(queryArgs)?email=rocky@redraccoon.org&password=rcky` (optional `&targetUrl=...` to land on a specific page; 302 + session cookie). To switch users: navigate `/rabid.logout()` first, then the GET login with the other tier's credentials. Demo suite: `rabid/browser_test_demo.ts` (intermixes in-process + in-browser checks; `TEST_RUNS` registry of named runs). Run it two ways: the page's "Run demo tests" button, or **`./rabid.sh test-run <name>`** (default `demo`) — starts the server, waits for a live client, runs the suite, prints a summary, exits 0/1 (2=unknown run, 3=no client). Edit→rerun loop: leave one browser tab open at the test-client page; re-run `./rabid.sh test-run …` after each code change — the tab **reconnects across the restart with no clicking** (opt-in is durable in the session row; `test-agent.js` re-polls; cmdId carries a per-process boot prefix so a reconnecting tab can't false-dedup the new server's first command). Only a **db rebuild** (wipes the session table) forces a re-login + re-open. (Robustness: `test-agent.js` treats a failed result-POST as a reconnect, not a crash — else a result racing the post-run shutdown would freeze the tab and break no-revisit.)

Eval endpoint (dev god-mode for exploring a wedged server/browser): `POST /eval {password, target:'server'|'browser', js, timeoutMs?}` → `{ok,value}`/`{ok:false,error}`. Matched on the raw path (not the route interpreter, so the posted code isn't parsed as a route). Authorised by a **separate** `eval-password` (`rabid-eval-password.txt`, 0600, gitignored) generated by startServer **only on a non-production db** — hard-off on production. `target:'server'` uses **direct `eval`** (not `new Function`, which sees only globals) wrapped in an async IIFE and run under `security.runSystem`, so the code reaches rabid.ts's lexical scope — `rabid`, all imports (`security`, `db`, `table`, `volunteer`, `event`, …), globals, and `await import(...)`; result JSON-safed via `serializeValue`. `target:'browser'` reuses `evalInBrowser`.

`alert()` is banned in client JS (it blocks the event loop and stalls automation): `tx()` uses `showAlert()` in `resources/rabid-scripts.js` — a non-blocking Bootstrap toast that records the message in `window.current_alert` while visible (a test can read it) and falls back to native `alert()` only if the toast never shows. showAlert also dedups: a message identical to one still on screen is not shown again.

**RPC storm watchdog** (2026-06, in `rpc()` in `resources/rabid-scripts.js`): twice (both times under interrupted puppeteer tooling, never a real user flow) ~3,100 tx calls stormed and blew the stack, destroying the evidence. `rpc()` now keeps an in-flight counter; past `window.lmTxMaxInFlight ?? 50` it captures ONE report to `window.lmTxStormReport` (tripping expr, mid-recursion stack = the cycle, ring buffer of last 20 call exprs) + `console.error`, then throws **synchronously** on every further call — `rpc` is deliberately a sync function returning the async core's promise, because a throw inside an `async function` only rejects the promise and cannot break synchronous recursion. If a storm recurs: read `window.lmTxStormReport` via puppeteer evaluate or `/eval`.
